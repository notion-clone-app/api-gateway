# Notion Clone — API Gateway 🚀

Микросервис гейтвея для платформы Notion Clone, написанный на Go. Отвечает за динамическую маршрутизацию запросов к внутренним сервисам, авторизацию, логирование и сбор OpenAPI документации.

## 🛠️ Системные требования

Перед началом работы убедитесь, что у вас установлены следующие инструменты:
* **Go** (версия 1.25+)
* **Docker** (с запущенным демоном)
* **Kind** (локальный Kubernetes кластер)
* **kubectl** (интерфейс командной строки K8s)
* **stern** (опционально, для удобных цветных логов) — устанавливается через `brew install stern` или `choco install stern`.

## ⚡ Быстрый старт в Kubernetes

Все основные команды обернуты в `Makefile`. Для просмотра всех доступных команд выполните:
```bash
make help
```

### 1. Подготовка окружения
Если у вас еще нет локального кластера Kubernetes, создайте его одной командой:
```bash
make cluster-up
```

### 2. Локальный деплой (Сборка + Запуск)
Чтобы автоматически собрать Docker-образ из текущих исходников Go, загрузить его в кластер `Kind` и применить K8s-манифесты, запустите:
```bash
make all
```
Команда сама дождется, пока поды перейдут в статус `Running`.

### 3. Проброс трафика на локальную машину
Так как гейтвей развернут внутри изолированной сети K8s, пробросьте порт на свой `localhost`:
```bash
make port-forward
```
После этого гейтвей станет доступен для Postman / cURL по адресу: **`http://localhost:8080`**.

### 4. Просмотр цветных логов
Чтобы видеть структурированные и цветные логи гейтвея в реальном времени (включая трассировку OpenTelemetry), откройте новое окно терминала и выполните:
```bash
make logs
```
*Примечание: Если утилита `stern` не установлена, команда автоматически переключится на стандартный вывод `kubectl logs`.*

## 💻 Варианты локального запуска

Проект поддерживает два режима работы в зависимости от ваших целей.

### Вариант 1: Быстрая разработка (через Docker Compose)
Идеально подходит для повседневного написания кода. Поддерживает монтирование кода «на лету» и включает в себя `mock-service` для тестирования роутинга.

```bash
# Запустить гейтвей и заглушку сервиса
make compose-up

# Посмотреть логи гейтвея
make compose-logs

# Остановить окружение
make compose-down
```
После запуска выполните тестовый запрос. Гейтвей автоматически распознает префикс `mock-service`, обратится к соседнему контейнеру и вернет ответ:
```bash
curl http://localhost:8080/api/v1/mock-service/hello
```

### Вариант 2: Интеграционная проверка (в Kubernetes Kind)
Используйте этот вариант перед оформлением Pull Request, чтобы убедиться, что манифесты, лимиты памяти, сетевые домены кластера и Graceful Shutdown работают корректно внутри K8s.

```bash
# 1. Создать кластер (если еще не создан)
make cluster-up

# 2. Собрать production-образ, загрузить в K8s и задеплоить манифесты
make all

# 3. Включить проброс портов и цветные логи в разных терминалах
make port-forward
make logs
```

## 🧹 Очистка ресурсов

Если вам нужно удалить гейтвей из кластера, выполните:
```bash
make clean
```

Для полного удаления локального кластера Kubernetes (освобождения оперативной памяти):
```bash
make cluster-down
```
